# AOS TCC Capability Broker Inventory V0

## 1. Executive Summary

The largest remaining Swift public-policy cluster is `src/commands/operator.swift`. It still owns public `ready`, `status`, `doctor`, and `permissions` grammar; text presentation; recovery choice; next-action generation; permission onboarding UI copy; LaunchAgent/runtime interpretation; and TCC reset sequencing. The native reasons for this code are real, but narrower than the public commands: Swift must expose permission facts, daemon/socket facts, LaunchAgent/native lifecycle facts, and permission prompt/reset actions. The public workflows built from those facts should move to external composition.

The only public `$AOS_PATH` route that should stay essentially native is `serve`, because it starts the AppKit/daemon/socket/TCC process identity. The next extraction order should be:

1. Add or formalize small private runtime/readiness fact primitives.
2. Move `ready` workflow and presentation to `scripts/aos-ready.mjs`.
3. Move `status` and `doctor` presentation to `scripts/aos-status.mjs` and `scripts/aos-doctor.mjs`.
4. Split permission workflow composition from native TCC probes, prompts, setup marker writes, and targeted reset actions.
5. Add guard tests so public policy, help, recovery text, and command grammar do not creep back into Swift.

No real external release boundary was found in the inspected repo state. The current callers, manifests, docs, and tests are in-repo and should cut over with no shims.

## 6. Cutover And No-Shim Decisions

In-repo callers/tests/docs that should be updated strictly during extraction:

- `manifests/commands/aos-external-commands.json` currently routes five public paths through `$AOS_PATH`/`__...`; `status`, `ready`, `doctor`, and `permissions` should cut over to scripts in one slice.
- `manifests/commands/aos-commands.json` owns public forms for `status`, `ready`, `doctor`, and `permissions`; those forms should remain canonical and Swift usage strings should disappear from public paths.
- `tests/external-command-dispatch.sh` already asserts manifest-backed dispatch for service and likely needs new assertions that the four cutover routes no longer use `$AOS_PATH`.
- Readiness/permission tests such as `tests/ready-ownership-mismatch.sh`, `tests/input-tap-readiness.sh`, `tests/input-tap-readiness-legacy.sh`, `tests/input-tap-readiness-classifier.sh`, and `tests/manual/tcc-reset-agent-user-path.sh` should be updated to assert script output and private primitive facts directly.
- `docs/dev/workflow-rules.json` includes `./aos ready` as a check; no shim is needed, but expected recommendation behavior should remain no-build for script/manifest changes.
- Compatibility pointers in `src/CLAUDE.md` are historical and should not block no-shim cutover.

No external release boundary was found. If a packaged app or published external CLI is later identified as consuming `__status`, `__ready`, `__doctor`, or `__permissions` directly, require an owner, removal gate, and evidence before adding any compatibility path.

## 7. Verification And Guard Ideas

- Manifest route guard: fail if public paths other than an allowlisted bootstrap set route to `$AOS_PATH` or use `argv_prefix` entries beginning with `__`. Initial allowlist should include `serve` and narrowly named private primitives only when invoked by external scripts.
- Help contract tests: ensure public usage and descriptions for `ready`, `status`, `doctor`, and `permissions` come from `manifests/commands/aos-commands.json` and not Swift `Usage:` strings.
- No-build policy test: `node scripts/aos-dev-workflow.mjs recommend --json --paths scripts/aos-ready.mjs manifests/commands/aos-external-commands.json manifests/commands/aos-commands.json` should recommend hot-swappable command-surface tests and not Swift build.
- Swift policy `rg` guard: allowlist private primitive diagnostics, then fail on new public route usage strings, recovery headings, `Next:`, `Human action needed:`, `Run './aos`, `service restart`, or public command grammar inside `src/commands/operator.swift`.
- Primitive schema tests: add fixtures for `__permissions facts`, `__daemon health`, `__runtime status-facts`, and legacy daemon health payloads.
- Cutover tests: assert `status`, `ready`, `doctor`, and `permissions` routes execute `/usr/bin/env node scripts/aos-*.mjs` in the external manifest and that no direct public manifest route points to `__status`, `__ready`, `__doctor`, or `__permissions`.
- Text/JSON fixture tests: preserve expected machine-readable fields while allowing public text to be owned by scripts.
