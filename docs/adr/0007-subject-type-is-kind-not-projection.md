# `subject_type` names the kind of a Subject, not a contextual projection

A Subject's `subject_type` is the stable kind it represents (`wiki.entity`, `sigil.agent`, `sigil.radial_menu.item_3d`, etc.) and does not change depending on which workbench, app, or entry path opened it. When a wiki page documents a domain concept like Sigil, the wiki document is its own Subject (`subject_type: wiki.entity`, source identity = wiki path) and the domain concept is a separate Subject (`subject_type: sigil.agent`); the domain Subject carries a **Subject Reference** to the wiki document Subject as the source of its narrative-Layer Facet. The same bytes back two stable Subject identities, explicit reference between them.

We rejected treating `subject_type` as a contextual projection (a wiki page "becoming" `sigil.agent` when opened through Sigil) because it makes `subject_type` an unreliable discriminator for capabilities, persistence, schema validation, verifier health, and Work Record correlation. We rejected embedding-style identity (one Subject swallowing another based on entry path) because two Work Records that touch the same file would appear unrelated unless every consumer understood the embedding trick. The `aos-workbench-pattern.md` "wiki page can project as ... `sigil.agent`" wording is amended; some current helpers/tests still emit app-specialized `subject_type` values from wiki paths and should be unwound as legacy projection behavior.
