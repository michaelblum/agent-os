# `subject_type` names the kind of a Subject, not a contextual projection

A Subject's `subject_type` is the stable kind it represents (`wiki.entity`, `service.runtime`, `artifact.bundle`, etc.) and does not change depending on which workbench, app, or entry path opened it. When a wiki page documents a domain concept, the wiki document is its own Subject (`subject_type: wiki.entity`, source identity = wiki path) and the domain concept is a separate Subject (`subject_type: service.runtime`); the domain Subject carries a **Subject Reference** to the wiki document Subject as the source of its narrative-Layer Facet. The same bytes back two stable Subject identities with an explicit reference between them.

We rejected treating `subject_type` as a contextual projection (a wiki page "becoming" a domain service when opened through a consumer) because it makes `subject_type` an unreliable discriminator for capabilities, persistence, schema validation, verifier health, and Work Record correlation. We rejected embedding-style identity (one Subject swallowing another based on entry path) because two Work Records that touch the same file would appear unrelated unless every consumer understood the embedding trick. Active helpers and tests keep wiki documents canonical and create separate domain Subjects explicitly.
