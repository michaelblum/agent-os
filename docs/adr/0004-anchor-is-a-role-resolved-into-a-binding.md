# Anchor is a role at the input grammar; resolution produces an Anchor Binding

A **Target** is an AOS scope address (`browser:`, `canvas:`, `screen:`, `ax:`); a **Ref** is a stable semantic element id within a Target's scope, when the dialect supports refs; a **Target-with-Ref** is the wire-form `<dialect>:<scope>/<ref>` that `aos see`/`aos do`/`aos show` operate on. **Anchor** is not a separate primitive — it is a *role* a Target-with-Ref plays when `aos show` uses it as a placement reference. The display subsystem resolves an input Anchor into an **Anchor Binding** that stores the lower-level placement state (e.g. `anchor_window + offset`) and lifecycle behavior, distinct from the original input string.

We rejected treating Anchor as synonymous with Target/Ref (loses the role-vs-address split visible in the wire format) and rejected treating Anchor as a fully separate top-level primitive (the input really is a Target-with-Ref; the extra metadata only appears post-resolution as the Anchor Binding). Today's CLI exposes `--anchor-browser`, `--anchor-window`, and `--anchor-channel` as concrete flags; a generic `--anchor <target>` flag may consolidate them later once those resolvers share one contract, but the plan should describe the role-vs-binding model now without misrepresenting the current command surface.
