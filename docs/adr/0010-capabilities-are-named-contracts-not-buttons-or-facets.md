# Capabilities are named contracts, distinct from Facets and Controls

A **Capability** on a Subject is a named contract the Subject promises to support — used by consumers (Subject Browsers, verifiers, exporters, replay tooling) to negotiate which behaviors the Subject implements. Capabilities are not button labels (those are **Controls**, derived) and not "what's available to open" (those are **Facets**, declared separately on the Subject). The v0 high-level Capability set is: `inspectable`, `editable`, `verifier-target`, `replayable`, `exportable`. Each has a documented contract in `shared/schemas/aos-subject-capabilities.md`. A Subject Browser uses `capabilities[]` to decide which behavior classes are safe to offer, then finds matching Facets to attach those behaviors to.

We rejected treating Capabilities as verbs (`open` / `edit` / `verify`) — too thin, gives consumers nothing to bind against — and rejected reusing them as Facet names (Facets already cover "what can I open"). The migration caveat: today's `capabilities[]` already contains operation/event-contract strings such as `markdown_document.text.patch`, `wiki.invoke`, `work_record.execution_map.edit`. These are useful operation contracts, not high-level Capability classes; the schema design pass should either document both shapes inside `capabilities[]` or split into `capabilities[]` (high-level) plus `contracts[]` (dotted operation strings). `navigable` was deliberately excluded from the v0 set — being an `aos.workbench.subject` already implies Subject Browsers can open it unless policy says otherwise; making it a Capability would invert the default.
